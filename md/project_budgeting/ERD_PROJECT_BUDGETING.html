Catatan Teknis

⚠️ Penting untuk Developer

  • Format ID SPK Budgeting — SPK-BUDGET/YY/MM/XXX (contoh: SPK-BUDGET/24/07/001)
  • Estimasi vs Final Pattern — Detail tables memiliki dual columns: _estimasi (dari penawaran/SPK original) dan _final (actual budgeting values). Memungkinkan perbandingan antara estimasi awal dan budget akhir.
  • Before/After/Result Pattern — Header table memiliki triple columns: _before (original dari SPK), _after (final budget), dan _result (selisih/difference).
  • Denormalisasi — Field nama (nm_customer, nm_project_leader, dll) disimpan langsung di tabel transaksi untuk menghindari JOIN berlebihan saat reporting.
  • Soft Delete — Pola delete_by + delete_date. NULL = aktif, terisi = dihapus.
  • Dual Database — Data employee dari database hr_sentral (koneksi dbhr), bukan dari db_consultant_new.
  • Related to SPK Penawaran & Penawaran — Budgeting terhubung ke kedua tabel induk melalui id_spk_penawaran dan id_penawaran.
  • No History Tables — Modul budgeting lebih sederhana dari SPK penawaran, tidak memiliki tabel history terpisah.
  • Single-level Approval — Hanya 1 level approval (langsung approve/reject), berbeda dengan SPK penawaran yang 6 level.
  • Nilai Kontrak Bersih — Dari SPK Penawaran, dihitung: nilai_kontrak - biaya_subcont - biaya_akomodasi - biaya_others - biaya_tandem - biaya_lab - biaya_subcont_tenaga_ahli - biaya_subcont_perusahaan.
