Alur Proses Approval SPK (Multi-level)

Flow approval SPK Penawaran dari input sampai fully approved/rejected (6 level)

Draft
sts_spk = NULL
→
Sales
approval_sales_sts
→
Konsultan 1*
approval_konsultan_1_sts
→
Konsultan 2*
approval_konsultan_2_sts
→
Project Leader
approval_project_leader_sts
→
Mgr Sales
approval_manager_sales
→
Direktur
approval_level2_sts
→
Approved
sts_spk = '1'
Any Level
→
Rejected
reject_xxx_sts + reason
→
sts_spk = '0'

* Konsultan 1 & 2 bersifat opsional, tergantung apakah id_konsultan_1 / id_konsultan_2 terisi.

Catatan Teknis

⚠️ Penting untuk Developer

  • Format ID SPK — XXX/STM/MKT-SPK/ROMAN_MONTH/YY (contoh: 001/STM/MKT-SPK/VII/24)
  • Format ID History — XXX/SPK-HIST/MM/YY (contoh: 001/SPK-HIST/07/24)
  • Multi-level Approval — 6 level approval: Sales → Konsultan 1 → Konsultan 2 → Project Leader → Manager Sales → Direktur. Konsultan 1 & 2 opsional.
  • Denormalisasi — Field nama (nm_customer, nm_sales, nm_project_leader, dll) disimpan langsung di tabel transaksi untuk menghindari JOIN berlebihan saat reporting.
  • Soft Delete — Pola deleted_by + deleted_date. NULL = aktif, terisi = dihapus.
  • Dual Database — Data employee dan divisi dari database hr_sentral (koneksi dbhr), bukan dari db_consultant_new.
  • Commission Tracking — Tracking komisi untuk pemberi informasi (max 2 orang) dan sales (max 2 orang), masing-masing dengan field persen + nominal.
  • Update Penawaran — Ketika SPK dibuat, field kons_tr_penawaran.id_spk_penawaran diupdate dengan ID SPK yang baru dibuat.
  • History Pattern — Setiap perubahan/revisi menghasilkan snapshot lengkap di tabel _history beserta semua detail (aktifitas, subcont, payment).
  • Nilai Kontrak Bersih — Dihitung dari: nilai_kontrak - biaya_subcont - biaya_akomodasi - biaya_others - biaya_tandem - biaya_lab - biaya_subcont_tenaga_ahli - biaya_subcont_perusahaan.
